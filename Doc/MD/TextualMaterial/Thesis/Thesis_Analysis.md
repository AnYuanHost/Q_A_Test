## 一、论文概述

本文研究如何结合胡塞尔意向性及其逻辑成果与现代计算机工程技术，构造非公理推理系统基础。重构胡塞尔意向性基本结构，设计更简单的表达结构，以此为基础展开胡塞尔意向性知识图谱模式，通过更松散复杂的知识表示来构建意向性世界。在工程设计上由基本集合论出发，以确保虚拟机的完备性和泛化能力，结合操作系统虚拟机技术并拓展轮询机制，构造基础系统保证理论知识图谱可以被解释和使用。

**关键词**：通用人工智能、非公理化推理系统、意向性、胡塞尔、知识图谱

## 二、理论路线背景（第1章）

### 2.1 三条AGI路线

| 路线 | 核心思想 | 代表成果 | 论文立场 |
|------|---------|---------|---------|
| **通用人工智能路线** | 哲学角度(认识论/意识/意向性) + 生产赋能 | 联结主义大模型平台化 | 弊端无法克服，不能实现真正通用 |
| **符号主义路线** | 专家系统→形式逻辑→描述逻辑→知识图谱 | Franz Baader描述逻辑体系[6] | 工具价值高但泛化不足，统计学导向 |
| **非公理逻辑路线** | 超越公理体系的推理引擎 | 王培NARS系统[7]、哲学引擎概念 | **本文选择路线**，但需理论重构 |

### 2.2 哲学引擎概念

论文提出"哲学引擎"方向作为非公理系统的发展思路：

- **引擎**：可以自由支撑各种推理逻辑的执行，保证完备性和泛化能力
- **自举**：管理功能交由内部推理进行，随系统迭代而调整，不受固定设计思路影响

### 2.3 核心论断

> "本文所完成的核心系统，完全独立于西方联结主义概念，率先完成新方向的成果转化。"

目标：逻辑外显于知识图谱，操作内化在虚拟机。

## 三、ESM框架设计理论（第2章 §2.3）

### 3.1 哲学基础

- **怀特海历程哲学**[24]：事物本质在于过程 → 映射的状态转移概念
- **功能主义问答观**[25]：任何程序执行 = 对某一问题的解答 → 操作记录为问题的解答
- **历程哲学重构失败**：直接应用需第四信息源解释背景，违背泛化性 → 工程折中方案

### 3.2 映射概念重构

原始映射概念（定义域→值域的对应关系）被重构为：

> 映射被描述为一种**行为**，其会引发定义域元素和值域元素的**状态转移**，而不是从定义域转移到值域。

- 定义域与值域重构为"真实存在所构建的域"，按**可读(定义域)**和**可写(值域)**划分
- 朴素希望：定义域元素为映射提供所需信息，共同导致值域元素发生改变

### 3.3 完备性证明

论证二元映射 `(Element& Domain, Element& Range)` 可以表达多元指令：

1. 提供序偶，第二元素存储索引信息
2. 通过读写方法将位置索引信息转移到映射中
3. 多元指令在语言中调用函数具备图灵完备
4. 只要多元信息能被初始化，多元指令即可得到执行

### 3.4 固定格式

| 结构 | 定义 | 工程对应 |
|------|-----|---------|
| **Element** | `{unsigned int Size, char* KeySpace}` | `E_S_M/h.cpp/E_S_M.h` |
| **Set** | `{Element SetParameter, SetTestFunction TestFunction}` | 同上 |
| **Map** | `{Set* Domain, Set* Range, MapFunction MapActive}` | 同上 |

运算符重载：

| 运算符 | 名称 | 语义 |
|--------|-----|------|
| `&=` | 转交传递 | 参数元素信息直接赋值，原元素归零 |
| `=` | 拷贝传递 | 深拷贝（较少使用） |
| `\|=` | 引用传递 | 原元素不归零 |

## 四、总元素环境检索器理论（第2章 §2.4）

### 4.1 核心定义

> 总元素环境检索器 = 独立智能体的**唯一元素感知能力**，划分一个智能体的参数空间。

- 精简传统符号表：舍弃关联信息 → 只提供检索功能
- 关联信息由知识库管理，不由符号表模块管理
- **多智能体基础**：多个EES各自独立 → 不同任务 → 协作

### 4.2 泛化语境设计

所有语境统一为固定格式，仅要求提供指令集并注册：

| 指令类别 | 指令内容 |
|---------|---------|
| 基本增删改查 | SubEESLogin, SubEESLogout, SubEESGetInf, SubEESPutInf, SubEESReadInf |
| 地址操作 | NewSubEESAddress, FirstSubEESAddress, NextSubEESAddress, ReleaseSubEESAddress |
| 超索引操作 | SubEESGetSuperIndex, SubEESPutSuperIndex, SubEESReadSuperIndex |
| 维护操作 | SubEESSort, SubEESRetain, SubEESDefragment, SubEESOutTest |
| 占用操作 | SubEESGetOccupy, SubEESPutOccupy |

### 4.3 三层映射架构

```
子检索器(泛化信息 → SuperIndex)
    → 超索引检索器(SuperIndex → 地址)
        → 元素环境(地址 → 实际元素)
```

- **子检索器种类**：字典序(LexOrder)、8比特(8Bit)、超索引(SuperIndex)
- **内部结构**：一级有序页表，懒处理注销，扩容页数 < 缩容保留页数（避免临界震荡）

### 4.4 工程对应

| 理论概念 | 代码模块 | 核心文件 |
|---------|---------|---------|
| 总元素环境检索器 | ESM_Manage | `EES_Base.h` |
| 元素环境 | ESM_Manage | `ElementEnvironment.h` |
| 字典序子检索器 | ESM_Manage | `SubEES_LexOrder.h` |
| 8比特子检索器 | ESM_Manage | `SubEES_8Bit.h` |

## 五、零轮询机理论（第2章 §2.5）

### 5.1 问答流图

定义：存储零轮询机应该解答的问题节点所组成的**有向图**，有向边标识节点间依赖关系。

节点核心信息：

- **定义域信息树** / **值域信息树**：叶子节点 = 序偶(语境信息, 泛化信息)
- **映射信息**：待执行的映射索引
- **前后置问题序列**：双向链接，方便后推/回收/碎片整理

### 5.2 四状态集合与状态转移

| 状态 | 变量名 | 含义 |
|------|--------|-----|
| 零状态 | QN_Zero | 初始化默认，非法状态 |
| 不可处理 | QN_Untractable | 前置问题未全部完成 |
| 待处理 | QN_Pending | 解决方案重点关注对象 |
| 可处理 | QN_Tractable | 应该被处理，等待轮询机分配 |
| 已处理 | QN_Processed | 由回收模块回收 |
| 等待 | QN_Wait | 轮询机执行中 |
| 成功 | QN_Success | 执行完成 |
| 可直接处理的不可处理 | QN_TraUntractable | 前置完成后跳过待处理直接进可处理 |

### 5.3 图操作

- **后推**：可处理集合中已完成节点 → 扩散影响后置节点状态
- **回收**：已处理集合内节点释放内存
- **碎片整理**：清理无效占用空间，注意地址变动需重连

### 5.4 内外循环

| 概念 | 含义 | 特点 |
|------|------|------|
| **内循环** | 代码中固定的维护逻辑 | 可舍弃问答流图格式，效率更高 |
| **外循环** | 被维护的问答流图 | 必须使用问答流图格式，逻辑执行在此完成 |
| **内循环管理** | 控制内循环泛化能力 | 可选取消以降低复杂度 |

### 5.5 轮询机

- 封装线程实现并行，独占处理一个问答节点
- 创建 → 连接节点 → 解释信息 → 组装可执行映射 → 线程执行 → 回收
- 加锁串行保证优先级，轮询机优先

### 5.6 工程对应

| 理论概念 | 代码模块 | 核心文件 |
|---------|---------|---------|
| 问答流图 | Polling_Base | `QuestionNode_Base.h` |
| 轮询机驱动 | Polling_Base | `PollingDriven_Base.h` |
| 零轮询机 | Polling_Base | `ZeroPollingDriven.h` |
| 管道 | Polling_Base | `PDZ_KRS_Pipe.h` |

## 六、解决方案理论（第2章 §2.6）

### 6.1 功能主义问答概念

> 解决方案 = 用于**子图替换**的信息，问答流图指定的子图使用该信息完成替换，且该行为有**解决问题的趋势**。

### 6.2 理想-实际问题模型

- **理想问题**：抽象的、逻辑的，暗含潜在特征
- **实际问题**：具体的求解过程，不关注潜在特征
- 即使过于简单的理想问题，也可导向相当多且复杂的实际问题
- 两者**永远不被视为同一个**

### 6.3 子图替换六元信息

| 序号 | 信息 | 作用 |
|------|------|------|
| 1 | 移除节点序列 | 析构并破坏指定节点 |
| 2 | 添加节点序列 | 注册新节点 |
| 3 | 前置问题后置重设序列 | 重建前节点连接 |
| 4 | 后置问题前置重设序列 | 重建后节点连接 |
| 5 | 注册元素序列 | 向EES提交注册 |
| 6 | 注销元素序列 | 向EES提交注销 |

### 6.4 后退问题与自举

- **后退问题**：每个辩护需要进一步辩护 → 解决方案工厂需要解决方案 → 无限回退
- **解决方案工厂**：识别 + 地址解释 + 信息解释 → 生成解决方案
- **解决方案自举构建**：固定方法集，仅面向工厂，阻断后退，不完全封闭接受迭代

| 理论概念 | 代码模块 | 核心文件 |
|---------|---------|---------|
| 解决方案 | Solution | `Solution.h` |
| 解决方案工厂 | Solution | `SolutionFactory.h` |

| 解决方案匹配 | Solution | `SolutionMatch.h` |

## 七、胡塞尔模型重构（第2章 §2.7）— 核心理论创新

### 7.1 原模型问题

胡塞尔原始意向性定义：

- **质性**（Qualität）：行为特征，决定行为类型
- **质料**（Materie）：行为内容，决定意向对象

问题：质料将对象内容和非对象内容整合，两者间无直接联系，且都和质性存在联系。

### 7.2 重构方案

| | 原模型 | 重构模型 |
|---|-------|---------|
| **质性** | 行为特征 | **对象 + 意向特征的融合产物** |
| **质料** | 内容 + 对象 | **仅非对象内容** |

关键解释：胡塞尔的"侧面"概念 — 意向性所意向的是实际对象的一个侧面，将侧面（理想环境下意向到的真实对象）与质性结合 → 新框架下的对象。

### 7.3 知识图谱模式对比

| 特征 | 传统知识图谱 | 意向性知识图谱 |
|------|------------|--------------|
| 连接方向 | 垂直领域连接（同领域内相关知识） | **横向跨领域连接**（质性跨领域结构连接） |
| 规则依赖 | 依赖给定规则 | **不依赖固定规则** |
| 检索方式 | 关系直接检索 | 质性可直接检索，质料需质性启发检索 |
| 统计导向 | 有 | 无 |
| 存储结构 | 三元组(主-谓-宾) | 意向性对象序列[序偶(质性,质料)] |

### 7.4 超越符号主义的特征

> "整个知识图谱几乎不受任何规则的要求，可以满足泛化的要求。这种泛化能力并不像传统符号主义应用那样依赖于某些给定的规则。"
>
> 结合虚拟机内核系统后 → 非公理推理 → 用户自主构建任何逻辑并执行

### 7.5 工程对应

| 理论概念 | 代码模块 | 核心文件 |
|---------|---------|---------|

## 八、意向性对象与知识图谱方法（第2章 §2.8-2.9）

### 8.1 意向性对象结构

- **序列**：子元素均为序偶，存储质性信息和质料信息
- **唯一索引方案**：SuperIndex统一有序分配，支持二分查找
- **泛化方案**：附属导向知识域信息辅助查询，支持跨语境检索

### 8.2 溯源器设计

核心知识图谱管理工具，维护有序序列记录访问路径：

1. **构建阶段**：问答流图调用相关知识构建有序序列
2. **获取阶段**：每个获取操作仅调用溯源器下一方法

分离构建与获取 → 减少维护成本

### 8.3 临时知识 vs 持久知识

| 类型 | 特征 | 索引方式 | 管理 |
|------|------|---------|------|
| **临时知识** | 运行时中介信息 | 唯一索引 | 临时知识索引管理器 |
| **持久知识** | 经确定化处理 | 唯一索引 + 泛化检索 | 管理模块分配 |

## 九、系统总架构（第3章 §3.6）

```
智能体 = 零轮询机 + 总元素环境检索器 + 知识库
```

- **零轮询机**：内循环管理外循环问答流图，轮询机管理+解决方案管理协作
- **总元素环境检索器**：符号表服务，子检索器→超索引→元素环境三层管理
- **知识库**：外挂于硬盘，装载到EES子检索器，通过知识图谱方法装卸载
- **通用映射集合**：可继承给下属零轮询机
- **私有映射集合**：单个零轮询机私有，不继承

### 运行流程

```
代码初始化 → 建立第一个工作环境
    → 零轮询机启动内循环
        → 内循环维护外循环问答流图
            → 解决方案管理模块执行子图替换
            → 轮询机管理模块分配轮询机
                → 轮询机解释信息 → 线程执行映射
                → 执行完成 → 后推 → 回收
    → 知识图谱生成新解决方案(并行)
```

## 十一、论文自述不足与展望（第5章）

### 不足

1. 知识图谱建设**成本极高**，前中期需大量人力辅助构建
2. 工程侧重理论实现，**性能优化**不足（页表管理过于简陋等）
3. 需建立共创平台和跨领域合作解决生态问题

### 展望

- 跨领域使用，建立足够庞大的知识体系维护生态
- 中心知识库跨过前中期后 → 辅助新知识生成和应用 → 实现通用人工智能工具理想
- 产业赋能，智库能力全面应用
